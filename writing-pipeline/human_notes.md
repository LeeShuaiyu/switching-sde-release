# Human Notes

This file is optional. Leave it blank if you want the pipeline to proceed from repository evidence alone.

If you do add notes, keep them short and factual. Good examples:

- what innovation to emphasize
- what problem the paper should foreground
- which result set is the final one if the repository contains many runs
- what style or venue family to target
- which sections to include beyond the experiment section
- terms to prefer or avoid
- known limitations that must be disclosed

Suggested template:

## Priority message

State the one idea the paper should leave with the reader.

## Target venue or style

Examples: IEEE-like, Nature-like, CCF-style CS venue, math or statistics style.

## Preferred notation or terminology

List symbols or field-standard terms you want used for the core objects.

## Enabled sections

- Experiment section: yes or no
- Introduction: yes or no
- Notation section: yes or no
- Problem formalization section: yes or no

## Must-show visuals

List the figures or tables that absolutely should appear in the final draft if they exist.

## Must emphasize

List the contributions, design choices, or problem settings that matter most.

## Must avoid

List claims, terms, or framings that would be inaccurate or strategically unhelpful.

## Repository hints

Point to the true final run, key figure, or decisive table if the repository is messy.
